Como gerir especificações de teste, execuções e medidas — TMF653
← Voltar ao Testing

Sobre este guia

Este guia explica como usar o módulo de Testing da TMF Service Layer, que implementa a API TMF:

  • TMF653 — Service Test Management: especificações de teste reutilizáveis (TestSpecification) e execuções de testes sobre instâncias de serviço (ServiceTest), com registo de medidas (TestMeasure)

Uma TestSpecification é um template reutilizável que define o tipo e os parâmetros de um teste. Um ServiceTest é uma execução concreta desse template contra uma instância de serviço no Inventário (TMF638). As medidas registadas durante a execução determinam se o teste passou ou falhou. Consulte o Guia de Aprovisionamento para perceber como activar serviços.

📋
Visão Geral — Relação entre Entidades
TestSpecification → ServiceTest → TestMeasure · integração com o Inventário

O módulo de Testing organiza-se em três níveis: a especificação (template), a execução (teste concreto) e as medidas (resultados capturados). Um serviço activo no Inventário é o alvo de cada execução.

ℹ️
A TestSpecification é opcional — é possível criar um ServiceTest sem associar uma especificação. Quando associada, a especificação deve estar em estado active ou retired (não obsolete).

TestSpecification — Especificações de Teste

📋
Estrutura de uma TestSpecification
Template reutilizável que define tipo, versão e parâmetros de teste

Uma TestSpecification é um template reutilizável que descreve o tipo de teste e os seus parâmetros padrão. Pode ser associada a uma ServiceSpecification do catálogo para indicar que os testes são aplicáveis a serviços desse tipo.

Campos da TestSpecification

CampoObrigatórioTipoDescrição
namesimstringNome descritivo da especificação (ex: "Connectivity Test Spec v1")
test_typenãostringTipo de teste: "functional", "performance", "regression", "smoke", etc.
versionnãostringVersão da especificação (ex: "1.0", "2.3")
descriptionnãostringDescrição livre dos objectivos e parâmetros do teste
service_spec_idnãouuidUUID de uma ServiceSpecification do catálogo (TMF633) a que esta especificação se aplica
↻
Ciclo de Vida da TestSpecification
active → retired → obsolete · eliminação só quando obsolete

Uma TestSpecification começa sempre no estado active. A progressão é unidireccional: active → retired → obsolete. Apenas especificações active podem ser usadas para criar novos testes. Apenas especificações obsolete podem ser eliminadas.

Caso de uso: criar e gerir uma especificação de connectividade

-- 1. Criar a especificação -- POST /tmf-api/serviceTest/v4/serviceTestSpecification { "name": "Connectivity Test Spec v1", "description": "Testa conectividade de rede end-to-end do serviço", "test_type": "functional", "version": "1.0" } → Resposta: { "id": "uuid", "lifecycle_status": "active", … } -- 2. Retirar a especificação (quando substituída por nova versão) -- PATCH /tmf-api/serviceTest/v4/serviceTestSpecification/<spec_id> { "lifecycle_status": "retired" } -- 3. Tornar obsoleta (arquivar definitivamente) -- PATCH /tmf-api/serviceTest/v4/serviceTestSpecification/<spec_id> { "lifecycle_status": "obsolete" } -- 4. Eliminar (só quando obsolete) -- DELETE /tmf-api/serviceTest/v4/serviceTestSpecification/<spec_id> → 204 No Content -- Listar especificações -- GET /tmf-api/serviceTest/v4/serviceTestSpecification
EstadoTransições PermitidasAceita Novos Testes?Pode ser Eliminada?
active retired sim não
retired obsolete não não
obsolete — (terminal) não sim

ServiceTest — Execuções de Teste

🧪
Estrutura de um ServiceTest
Execução concreta de um teste sobre uma instância de serviço

Um ServiceTest representa a execução de um teste sobre um serviço específico do inventário. Começa no estado planned, passa a inProgress quando arranca, e termina em completed, failed ou cancelled. As TestMeasures — as medições individuais capturadas durante o teste — só podem ser adicionadas enquanto o teste está inProgress.

Campos do ServiceTest

CampoObrigatórioTipoDescrição
namesimstringNome descritivo do teste (ex: "Connectivity Test — vm-01")
service_idsimuuidUUID do serviço alvo no inventário (TMF638)
modenãoenumautomated | manual | hybrid
descriptionnãostringDescrição livre do objectivo desta execução
test_specificationnãouuidUUID de uma TestSpecification não-obsolete
start_date_timenãodatetimeMomento de início efectivo do teste
end_date_timenãodatetimeMomento de conclusão do teste

Campos de uma TestMeasure

CampoObrigatórioTipoDescrição
metric_namesimstringIdentificador da métrica medida (ex: "latency_ms", "packet_loss_pct")
metric_valuenãofloatValor numérico capturado (ex: 42.5)
unitnãostringUnidade de medida (ex: "ms", "%", "Mbps")
resultsimenumpass | fail | inconclusive

Resultados de medida disponíveis

pass fail inconclusive

Modos de execução disponíveis

automated — teste executado por sistema automatizado sem intervenção humana

manual — teste executado manualmente por um operador

hybrid — combinação de execução automática com validação humana

↻
Ciclo de Vida do ServiceTest
planned → inProgress → completed | failed | cancelled · eliminação só em estado terminal

Um ServiceTest começa no estado planned. Ao ser iniciado, transita para inProgress — único estado em que podem ser adicionadas medidas (TestMeasure). A conclusão pode ser completed (sucesso), failed (falha) ou cancelled (cancelado). Todos os estados terminais permitem a eliminação do registo.

Caso de uso: executar um teste de conectividade

-- 1. Criar o teste (planned) -- POST /tmf-api/serviceTest/v4/serviceTest { "name": "Connectivity Test — vm-alpha-001", "description": "Verificação de conectividade de rede end-to-end", "service_id": "<uuid-do-serviço>", "mode": "automated" } → Resposta: { "id": "uuid", "state": "planned", … } -- 2. Iniciar o teste → inProgress -- PATCH /tmf-api/serviceTest/v4/serviceTest/<test_id> { "state": "inProgress", "start_date_time": "2026-03-08T10:00:00Z" } -- 3. Adicionar medidas (só possível quando inProgress) -- POST /tmf-api/serviceTest/v4/serviceTest/<test_id>/testMeasure { "metric_name": "latency_ms", "metric_value": 18.4, "unit": "ms", "result": "pass" } POST /tmf-api/serviceTest/v4/serviceTest/<test_id>/testMeasure { "metric_name": "packet_loss_pct", "metric_value": 0.1, "unit": "%", "result": "pass" } -- 4. Concluir o teste → completed -- PATCH /tmf-api/serviceTest/v4/serviceTest/<test_id> { "state": "completed", "end_date_time": "2026-03-08T10:05:00Z" } -- Alternativa: marcar como falha -- PATCH /tmf-api/serviceTest/v4/serviceTest/<test_id> { "state": "failed", "end_date_time": "2026-03-08T10:05:00Z" } -- Eliminar o teste (só em estado terminal) -- DELETE /tmf-api/serviceTest/v4/serviceTest/<test_id> → 204 No Content
⚠️
Medidas (TestMeasure) só podem ser adicionadas enquanto o teste está em inProgress. Tentativas de adicionar medidas em testes planned, completed, failed ou cancelled devolvem HTTP 409 Conflict.
EstadoTransições PermitidasAceita Medidas?Pode ser Eliminado?
planned inProgress, cancelled não não
inProgress completed, failed, cancelled sim não
completed — (terminal) não sim
failed — (terminal) não sim
cancelled — (terminal) não sim

Guia Passo a Passo — Cenário Completo

1
Criar uma TestSpecification
TMF653 POST /serviceTestSpecification  estado inicial: active

Cenário: criar uma especificação reutilizável para testes de conectividade que serão executados periodicamente sobre serviços de rede.

POST /tmf-api/serviceTest/v4/serviceTestSpecification { "name": "Network Connectivity Test Spec", "description": "Testa conectividade end-to-end: latência, perda de pacotes e throughput", "test_type": "functional", "version": "1.0" } → { "id": "spec-uuid", "lifecycle_status": "active", … }
✅
Na interface: use o separador Test Specifications e clique em + Create Specification. A especificação fica disponível imediatamente para associação a novos testes.
2
Criar e Iniciar um ServiceTest
TMF653 POST /serviceTest  planned → inProgress

Cenário: executar a especificação de conectividade sobre o serviço vm-alpha-001 do inventário.

-- 1. Criar o teste (planned) associado à especificação -- POST /tmf-api/serviceTest/v4/serviceTest { "name": "Connectivity Test — vm-alpha-001", "service_id": "<uuid-do-serviço>", "mode": "automated", "test_specification": "<spec-uuid>" } → { "id": "test-uuid", "state": "planned", … } -- 2. Iniciar o teste -- PATCH /tmf-api/serviceTest/v4/serviceTest/<test-uuid> { "state": "inProgress", "start_date_time": "2026-03-08T10:00:00Z" } → { "state": "inProgress", "start_date_time": "2026-03-08T10:00:00Z", … }
3
Adicionar TestMeasures ao Teste em Execução
TMF653 POST /serviceTest/{id}/testMeasure  apenas quando inProgress

Com o teste em inProgress, registam-se as medidas individuais capturadas durante a execução. Cada medida representa uma métrica verificada e o seu resultado. É possível adicionar múltiplas medidas para cobrir diferentes aspectos do teste.

-- Medida 1: latência (pass) -- POST /tmf-api/serviceTest/v4/serviceTest/<test-uuid>/testMeasure { "metric_name": "latency_ms", "metric_value": 18.4, "unit": "ms", "result": "pass" } -- Medida 2: perda de pacotes (pass) -- POST /tmf-api/serviceTest/v4/serviceTest/<test-uuid>/testMeasure { "metric_name": "packet_loss_pct", "metric_value": 0.1, "unit": "%", "result": "pass" } -- Medida 3: throughput (fail — abaixo do mínimo esperado) -- POST /tmf-api/serviceTest/v4/serviceTest/<test-uuid>/testMeasure { "metric_name": "throughput_mbps", "metric_value": 42.0, "unit": "Mbps", "result": "fail" } -- Listar medidas do teste -- GET /tmf-api/serviceTest/v4/serviceTest/<test-uuid>/testMeasure
ℹ️
Na interface, as medidas aparecem numa linha expansível abaixo de cada teste. Clique em ▶ Measures para expandir e no botão + Add Measure para adicionar uma nova medida enquanto o teste está inProgress.
4
Concluir o Teste e Consultar Resultados
TMF653 PATCH + GET  inProgress → completed | failed | cancelled

Após recolher todas as medidas, transitar o teste para o estado final adequado. Utilize completed quando o teste correu conforme esperado, failed se foram detectadas falhas críticas ou cancelled se o teste foi interrompido antes de concluir.

-- Concluir com sucesso -- PATCH /tmf-api/serviceTest/v4/serviceTest/<test-uuid> { "state": "completed", "end_date_time": "2026-03-08T10:08:00Z" } -- Concluir com falha -- PATCH /tmf-api/serviceTest/v4/serviceTest/<test-uuid> { "state": "failed", "end_date_time": "2026-03-08T10:08:00Z" } -- Cancelar o teste (de planned ou inProgress) -- PATCH /tmf-api/serviceTest/v4/serviceTest/<test-uuid> { "state": "cancelled" } -- Listar todos os testes -- GET /tmf-api/serviceTest/v4/serviceTest GET /tmf-api/serviceTest/v4/serviceTest?state=inProgress GET /tmf-api/serviceTest/v4/serviceTest?state=failed GET /tmf-api/serviceTest/v4/serviceTest?service_id=<uuid> -- Eliminar teste (só em estado terminal) -- DELETE /tmf-api/serviceTest/v4/serviceTest/<test-uuid> → 204 No Content
ℹ️
Todos os endpoints de listagem suportam paginação via limit e offset. Os cabeçalhos X-Total-Count e X-Result-Count são devolvidos em conformidade com o padrão TMF.
✔
Resumo — Fluxo Completo de Testing
Caso de uso típico — testar conectividade de um serviço em produção
  • 1 Verificar que existe um Service activo no Inventário (TMF638) — obter o service_id
  • 2 (Opcional) Criar uma TestSpecification reutilizável via POST /serviceTestSpecification com o tipo e versão do teste
  • 3 Criar um ServiceTest via POST /serviceTest com service_id e, opcionalmente, o ID da especificação
  • 4 Iniciar o teste via PATCH state: "inProgress" — a partir deste momento é possível adicionar medidas
  • 5 Adicionar TestMeasures via POST /serviceTest/{id}/testMeasure para cada métrica capturada
  • 6 Concluir o teste via PATCH state: "completed" (ou "failed" se relevante)
  • 7 Quando uma especificação for substituída por nova versão, movê-la para retired e depois obsolete
  • 8 Eliminar testes em estado terminal para manter o dashboard limpo
