Problem Management — Help
Step-by-step guide for TMF621 Trouble Tickets and TMF656 Service Problems
← Back to Problems
This guide covers the full lifecycle for Trouble Tickets (TMF621) and Service Problems (TMF656). Both entities follow a structured state machine and can be linked to Service Inventory instances and/or Service Assurance Alarms.
1
Create a Trouble Ticket

A Trouble Ticket captures a reported service problem from a customer or operator. It can be linked to a Service Inventory instance (related_service_id) and/or an active Alarm (related_alarm_id).

Click + New Ticket on the Problems page and complete the form. Only name is mandatory. A new ticket always starts in state submitted.

POST /tmf-api/troubleTicketManagement/v4/troubleTicket
{
  "name": "Customer 12345 – Internet Down",
  "description": "Customer reports no connectivity since 08:00 UTC",
  "severity": "critical",
  "priority": 1,
  "ticket_type": "serviceFailure",
  "related_service_id": "<uuid>",
  "related_alarm_id": "<uuid or omit>",
  "notes": [{ "text": "Opened by NOC shift B", "author": "NOC-B" }]
}
2
Trouble Ticket State Machine

Use the State → button on any ticket row to advance the state. Invalid transitions are rejected with HTTP 422.

FromAllowed transitions to
submittedinProgress
inProgresspending   resolved
pendinginProgress   resolved
resolvedclosed
closedterminal — no further transitions

When transitioning to resolved, an optional resolution text can be provided. The server automatically sets resolution_date.

PATCH /tmf-api/troubleTicketManagement/v4/troubleTicket/{id}
{ "state": "resolved", "resolution": "Faulty ONT replaced on-site" }
3
Manage Notes on a Ticket

Notes are append-only work-log entries. Open a ticket's Detail panel and use + Add Note. Notes can also be deleted if incorrect.

# Add a note
POST /tmf-api/troubleTicketManagement/v4/troubleTicket/{ticket_id}/note
{ "text": "Customer confirmed service restored.", "author": "NOC-A" }

# List notes
GET /tmf-api/troubleTicketManagement/v4/troubleTicket/{ticket_id}/note

# Delete a note
DELETE /tmf-api/troubleTicketManagement/v4/troubleTicket/{ticket_id}/note/{note_id}
4
Create a Service Problem

A Service Problem (TMF656) is a higher-level record tracking root cause analysis for one or more trouble tickets. Switch to the Service Problems tab and click + New Problem.

A Service Problem can be linked to a related_service_id and an optional related_ticket_id. It always starts in state submitted.

POST /tmf-api/serviceProblemManagement/v4/problem
{
  "name": "Core Router BGP Flapping — Datacenter A",
  "impact": "criticalSystemImpact",
  "priority": 1,
  "category": "networkFailure",
  "related_service_id": "<uuid>",
  "related_ticket_id": "<uuid or omit>"
}
5
Service Problem State Machine

Service Problems follow a slightly different lifecycle that includes a confirmation step and a rejection terminal state.

FromAllowed transitions to
submittedconfirmed   rejected
confirmedactive   rejected
activeresolved
resolvedclosed
rejectedterminal
closedterminal
 
PATCH /tmf-api/serviceProblemManagement/v4/problem/{id}
{ "state": "confirmed" }

PATCH /tmf-api/serviceProblemManagement/v4/problem/{id}
{ "state": "resolved", "resolution": "BGP route reflector reconfigured" }
6
Filtering and Querying

Both resources support query parameter filtering. Use the filter dropdowns on the Problems page or query directly.

# Trouble Tickets
GET /tmf-api/troubleTicketManagement/v4/troubleTicket?state=inProgress&severity=critical
GET /tmf-api/troubleTicketManagement/v4/troubleTicket?related_service_id=<uuid>

# Service Problems
GET /tmf-api/serviceProblemManagement/v4/problem?state=active&impact=criticalSystemImpact
GET /tmf-api/serviceProblemManagement/v4/problem?related_service_id=<uuid>

Pagination is supported with offset and limit query parameters (default limit: 100).

7
Referential Integrity Rules

The following FK constraints are enforced by the API:

FieldEntityConstraint
related_service_idTroubleTicketService must exist (RESTRICT on delete)
related_alarm_idTroubleTicketAlarm must exist; set to NULL if alarm deleted
related_service_idServiceProblemService must exist (RESTRICT on delete)
related_ticket_idServiceProblemTroubleTicket must exist; set to NULL if ticket deleted

If a referenced entity does not exist, the API returns HTTP 404.

8
Integration with Assurance Alarms

From the Assurance page you can navigate directly to the Problems page with a pre-filled alarm ID. Use the deep-link shortcut from any alarm row to open the "New Ticket" modal with the alarm already populated:

problems.html?alarm_id=<alarm-uuid>

This creates a direct Alarm → Trouble Ticket relationship, enabling end-to-end incident tracing from the first alert to final resolution and closure.
