❓ Guia de Problem Management
Trouble Tickets e Service Problems — TMF621 · TMF656
← Voltar ao Problems

Sobre este guia

Este guia explica como usar o módulo de Problem Management da TMF Service Layer, que implementa duas APIs TMF:

  • TMF621 — Trouble Ticket Management: tickets reportados por clientes ou operadores, ligados a serviços e/ou alarmes
  • TMF656 — Service Problem Management: problemas de serviço de nível superior, consolidando root cause analysis de um ou mais tickets

Ambos os recursos podem referenciar uma instância de serviço (related_service_id) do Inventário (TMF638). Os alarmes do módulo de Assurance podem ser ligados directamente a Trouble Tickets via related_alarm_id.

⚠
Pré-requisito — Serviço no Inventário
Tickets e Problems podem ser associados a um service_id do Inventário (TMF638)

Embora related_service_id seja opcional nos formulários, é boa prática sempre associar um ticket ou problem a um serviço afectado do inventário, facilitando a rastreabilidade end-to-end.

ℹ️
Ainda não tem dados? Use a secção Gerar Dados de Teste no final desta página para criar automaticamente tickets e problems de exemplo.

TMF621 — Trouble Ticket Management

1
Criar um Trouble Ticket
Reportar uma falha ou degradação — clique em "+ New Ticket"

Um Trouble Ticket captura um problema reportado por um cliente ou operador. Pode ser ligado a uma instância de serviço (related_service_id) e/ou a um alarme activo (related_alarm_id). Apenas name é obrigatório. O ticket inicia sempre no estado submitted.

POST /tmf-api/troubleTicketManagement/v4/troubleTicket { "name": "Cliente 12345 – Internet Down", "description": "Cliente sem conectividade desde as 08:00 UTC", "severity": "critical", "priority": 1, "ticket_type": "serviceFailure", "related_service_id": "<uuid>", "related_alarm_id": "<uuid ou omitir>", "notes": [{ "text": "Aberto pelo turno B do NOC", "author": "NOC-B" }] }
2
Máquina de Estados — Trouble Ticket
Ciclo de vida: submitted → inProgress → resolved → closed

Use o botão State → em cada linha de ticket para avançar o estado. Transições inválidas são rejeitadas com HTTP 422.

EstadoTransições permitidas
submittedinProgress
inProgresspending   resolved
pendinginProgress   resolved
resolvedclosed
closedterminal — sem mais transições
ℹ️
Ao transitar para resolved, forneça resolution com o texto da resolução. O servidor define automaticamente resolution_date.
PATCH /tmf-api/troubleTicketManagement/v4/troubleTicket/{id} { "state": "resolved", "resolution": "ONT defeituosa substituída no local" }
3
Gerir Notas de um Ticket
Entradas de work-log append-only associadas a um ticket

As notas são entradas de work-log append-only. Abra o painel Detail de um ticket e use + Add Note. Notas erradas podem ser eliminadas individualmente.

# Adicionar nota POST /tmf-api/troubleTicketManagement/v4/troubleTicket/{ticket_id}/note { "text": "Cliente confirmou restauro do serviço.", "author": "NOC-A" } # Listar notas GET /tmf-api/troubleTicketManagement/v4/troubleTicket/{ticket_id}/note # Eliminar nota DELETE /tmf-api/troubleTicketManagement/v4/troubleTicket/{ticket_id}/note/{note_id}

TMF656 — Service Problem Management

4
Criar um Service Problem
Registo de análise de causa raiz para um ou mais tickets

Um Service Problem (TMF656) é um registo de nível superior que agrega a análise de causa raiz para um ou mais trouble tickets. Mude para o separador Service Problems e clique em + New Problem.

POST /tmf-api/serviceProblemManagement/v4/problem { "name": "Core Router BGP Flapping — Datacenter A", "impact": "criticalSystemImpact", "priority": 1, "category": "networkFailure", "related_service_id": "<uuid>", "related_ticket_id": "<uuid ou omitir>" }
5
Máquina de Estados — Service Problem
Ciclo com confirmação obrigatória: submitted → confirmed → active → resolved → closed

Os Service Problems incluem um passo de confirmação e o estado terminal rejected.

EstadoTransições permitidas
submittedconfirmed   rejected
confirmedactive   rejected
activeresolved
resolvedclosed
rejectedterminal
closedterminal
PATCH /tmf-api/serviceProblemManagement/v4/problem/{id} { "state": "confirmed" } PATCH /tmf-api/serviceProblemManagement/v4/problem/{id} { "state": "resolved", "resolution": "Route reflector BGP reconfigurado" }
6
Fluxo End-to-End Típico
Alarme → Ticket → Problem → Resolução
  • 1 No Assurance, um alarme é levantado (raised) para um serviço degradado
  • 2 Na página Problems, criar um Trouble Ticket com related_alarm_id (ou via link directo problems.html?alarm_id=…)
  • 3 Avançar o ticket para inProgress e adicionar notas de work-log
  • 4 Se o problema for recorrente, criar um Service Problem ligando-o via related_ticket_id
  • 5 Avançar o Problem por confirmed → active → resolved → closed
  • 6 Resolver e fechar o Ticket; reconhecer e limpar o Alarme no Assurance
7
Filtros e Integridade Referencial
Parâmetros de pesquisa e constraints FK
# Trouble Tickets GET /tmf-api/troubleTicketManagement/v4/troubleTicket?state=inProgress&severity=critical GET /tmf-api/troubleTicketManagement/v4/troubleTicket?related_service_id=<uuid> # Service Problems GET /tmf-api/serviceProblemManagement/v4/problem?state=active&impact=criticalSystemImpact GET /tmf-api/serviceProblemManagement/v4/problem?related_service_id=<uuid>
CampoEntidadeConstraint
related_service_idTroubleTicketServiço deve existir (RESTRICT on delete)
related_alarm_idTroubleTicketAlarme deve existir; SET NULL se alarme eliminado
related_service_idServiceProblemServiço deve existir (RESTRICT on delete)
related_ticket_idServiceProblemTicket deve existir; SET NULL se ticket eliminado
⚠️
Se uma entidade referenciada não existir, a API retorna HTTP 404.

  7. Avança o Problem para confirmed

Execuções repetidas: cada execução cria conjuntos de recursos independentes com sufixo numérico (ex: "svc-problems-demo-2").
