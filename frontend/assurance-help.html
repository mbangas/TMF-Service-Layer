❓ Guia de Assurance de Serviços
Como gerir alarmes, medições e SLOs — TMF642 · TMF628 · TMF657
← Voltar ao Assurance

Sobre este guia

Este guia explica como usar o módulo de Assurance da TMF Service Layer, que agrega três APIs TMF:

  • TMF642 — Alarm Management: gestão de alarmes associados a serviços (raised → acknowledged → cleared)
  • TMF628 — Performance Management: medições de métricas de desempenho (scheduled → completed/failed)
  • TMF657 — Service Level Management: objectivos de nível de serviço (SLOs) com monitorização de violações (active ↔ violated ↔ suspended)

Todos os recursos referenciam uma instância de serviço (service_id) do Inventário (TMF638). Os serviços devem existir antes de criar alarmes, medições ou SLOs. Consulte o Guia de Aprovisionamento para perceber como activar serviços.

⚠
Pré-requisito — Serviço no Inventário
Alarmes, medições e SLOs requerem um service_id válido do inventário (TMF638)

Todos os recursos de Assurance associam-se a uma instância de Service no inventário através do campo service_id (obrigatório). O serviço pode estar em qualquer estado — não é necessário que esteja active para criar alarmes ou medições, embora na prática se monitorizem tipicamente serviços activos.

ℹ️
Ainda não tem serviços? Use a secção Gerar Dados de Teste no final desta página para criar automaticamente um serviço e exemplos de alarmes, medições e SLOs.

TMF642 — Gestão de Alarmes

🔔
Estrutura de um Alarme
Campos, severidades e ciclo de vida raised → acknowledged → cleared

Um Alarm representa um evento de falha ou degradação detectado num serviço. Cada alarme tem uma severidade, uma causa provável e percorre um ciclo de vida linear desde que é levantado (raised) até ser resolvido (cleared).

Campos do Alarm

CampoObrigatórioTipoDescrição
namesimstringNome descritivo do alarme (ex: "Alto CPU vm-alpha")
service_idsimuuidUUID do serviço afectado no inventário (TMF638)
descriptionnãostringDescrição livre do evento de alarme
alarm_typenãostringClassificação do tipo de alarme (ex: "processingErrorAlarm", "equipmentAlarm")
severitynãoenumcritical | major | minor | warning | indeterminate
probable_causenãostringCausa provável (ex: "softwareError", "hardwareFailure")
specific_problemnãostringDescrição detalhada do problema detectado
raised_atnãodatetimeMomento em que o alarme foi levantado (default: hora de criação)

Severidades disponíveis

critical major minor warning indeterminate
↻
Ciclo de Vida do Alarme
raised → acknowledged → cleared · eliminação só quando cleared

Um alarme começa sempre no estado raised (levantado). A primeira transição é o reconhecimento (acknowledged), que confirma que um operador tomou conhecimento do problema. A resolução efectiva é marcada com cleared — único estado a partir do qual o alarme pode ser eliminado.

Caso de uso: gerir um alarme de CPU elevado

-- 1. Criar o alarme -- POST /tmf-api/alarmManagement/v4/alarm { "name": "Alto CPU — vm-alpha-001", "description": "Utilização de CPU superior a 95% durante 5 minutos", "service_id": "<uuid-do-serviço>", "alarm_type": "processingErrorAlarm", "severity": "critical", "probable_cause": "softwareError", "specific_problem":"CPU usage 97.3%% — processo webapp em loop" } → Resposta: { "id": "uuid", "state": "raised", … } -- 2. Reconhecer o alarme (operador notificado) -- PATCH /tmf-api/alarmManagement/v4/alarm/<alarm_id> { "state": "acknowledged" } → { "state": "acknowledged", "acknowledged_at": "2026-03-08T14:32:10Z" } -- 3. Resolver o alarme após correcção -- PATCH /tmf-api/alarmManagement/v4/alarm/<alarm_id> { "state": "cleared" } → { "state": "cleared", "cleared_at": "2026-03-08T15:10:00Z" } -- 4. Eliminar o alarme (só quando cleared) -- DELETE /tmf-api/alarmManagement/v4/alarm/<alarm_id> → 204 No Content -- Listar alarmes por severidade -- GET /tmf-api/alarmManagement/v4/alarm?severity=critical GET /tmf-api/alarmManagement/v4/alarm?state=raised&limit=50
⚠️
Apenas alarmes no estado cleared podem ser eliminados. Tentativas de eliminar alarmes em raised ou acknowledged devolvem HTTP 409 Conflict.
EstadoTransições PermitidasPode ser Eliminado?
raisedacknowledgednão
acknowledgedclearednão
cleared— (terminal)sim

TMF628 — Gestão de Desempenho

📊
Medições de Desempenho
Recolha de métricas por serviço — scheduled → completed / failed

Uma PerformanceMeasurement representa a recolha de uma métrica específica de um serviço (latência, throughput, disponibilidade, etc.). Cada medição tem um nome de métrica (metric_name), uma granularidade temporal e um valor preenchido quando a medição é concluída.

Campos da PerformanceMeasurement

CampoObrigatórioTipoDescrição
namesimstringNome descritivo (ex: "Latência média — vm-alpha Março 2026")
service_idsimuuidUUID do serviço alvo no inventário (TMF638)
metric_namesimstringIdentificador da métrica (ex: "latency_ms", "throughput_mbps", "availability_pct")
descriptionnãostringDescrição livre da medição
metric_valuenãofloatValor medido — preenchido ao completar via PATCH
unit_of_measurenãostringUnidade da métrica (ex: "ms", "Mbps", "%")
granularitynãoenumminutely | hourly | daily | weekly | monthly
scheduled_atnãodatetimeQuando a medição foi agendada/efectuada

Caso de uso: recolher a latência de um serviço

-- 1. Criar a medição -- POST /tmf-api/performanceManagement/v4/performanceMeasurement { "name": "Latência média — vm-alpha Março 2026", "description": "Latência RTT média do serviço web em produção", "service_id": "<uuid-do-serviço>", "metric_name": "latency_ms", "unit_of_measure": "ms", "granularity": "hourly", "scheduled_at": "2026-03-08T10:00:00Z" } → Resposta: { "id": "uuid", "state": "scheduled", … } -- 2. Registar o valor medido e concluir -- PATCH /tmf-api/performanceManagement/v4/performanceMeasurement/<id> { "state": "completed", "metric_value": 12.4, "completed_at": "2026-03-08T10:05:00Z" } → { "state": "completed", "metric_value": 12.4, … } -- Marcar como falha (em caso de erro de recolha) -- PATCH /tmf-api/performanceManagement/v4/performanceMeasurement/<id> { "state": "failed" } -- Listar medições -- GET /tmf-api/performanceManagement/v4/performanceMeasurement?state=scheduled GET /tmf-api/performanceManagement/v4/performanceMeasurement?service_id=<uuid>
ℹ️
Os estados completed e failed são terminais — não é possível retransitar. Medições nos estados terminais podem ser eliminadas via DELETE.

TMF657 — Objectivos de Nível de Serviço (SLO)

📈
ServiceLevelObjective — Estrutura e Monitorização
SLOs monitorizam métricas contra um threshold e detectam violações automaticamente

Um ServiceLevelObjective (SLO) define um objectivo de qualidade para uma métrica específica de um serviço: por exemplo, "a latência deve ser inferior a 50 ms" ou "a disponibilidade deve estar acima de 99,9%". O SLO monitoriza continuamente o valor da métrica e transita automaticamente para violated quando a condição é incumprida.

Campos do ServiceLevelObjective

CampoObrigatórioTipoDescrição
namesimstringNome descritivo do SLO (ex: "Latência SLO — vm-alpha Produção")
service_idsimuuidUUID do serviço monitorizado no inventário (TMF638)
metric_namesimstringMétrica monitorizada (ex: "latency_ms", "availability_pct", "throughput_mbps")
descriptionnãostringDescrição livre do objectivo
threshold_valuenãofloatValor limite que define o objectivo
directionnãoenum"above" (alerta quando < threshold) ou "below" (alerta quando > threshold)
tolerancenãofloatTolerância aceitável em torno do threshold antes de reportar violação
sls_idnãouuidUUID da ServiceLevelSpecification do catálogo (TMF633) que este SLO implementa
↻
Ciclo de Vida do SLO e Detecção de Violações
active ↔ violated (auto) ↔ suspended (manual)

O SLO começa no estado active (a monitorizar). A transição para violated é automática — desencadeada pelo endpoint POST /serviceLevel/check_violations quando a métrica reportada ultrapassa o threshold. A transição para suspended é manual (via PATCH) e permite pausar a monitorização temporariamente.

Caso de uso: monitorizar a latência de um serviço

-- 1. Criar o SLO -- POST /tmf-api/serviceLevelManagement/v4/serviceLevel { "name": "Latência SLO — vm-alpha", "description": "Latência RTT não deve exceder 50ms", "service_id": "<uuid-do-serviço>", "metric_name": "latency_ms", "threshold_value":50.0, "direction": "below", "tolerance": 5.0 } → Resposta: { "id": "uuid", "state": "active", … } -- 2. Verificar violações (fornecer o valor actual da métrica) -- POST /tmf-api/serviceLevelManagement/v4/serviceLevel/check_violations { "service_id": "<uuid-do-serviço>", "metric_name": "latency_ms", "metric_value": 78.2 } → SLOs em 'active' com latency_ms / direction "below" / threshold 50: 78.2 > 50 + 5 (tolerância) → transição automática para 'violated' -- 3. Repor o SLO para active após correcção -- PATCH /tmf-api/serviceLevelManagement/v4/serviceLevel/<slo_id> { "state": "active" } -- 4. Suspender temporariamente (ex: janela de manutenção) -- PATCH /tmf-api/serviceLevelManagement/v4/serviceLevel/<slo_id> { "state": "suspended" } -- Eliminar SLO (só quando suspended) -- DELETE /tmf-api/serviceLevelManagement/v4/serviceLevel/<slo_id> -- Listar SLOs por estado -- GET /tmf-api/serviceLevelManagement/v4/serviceLevel?state=violated GET /tmf-api/serviceLevelManagement/v4/serviceLevel?service_id=<uuid>
🔴
A transição para violated é exclusivamente automática via POST /check_violations. Um PATCH manual para violated é rejeitado pela API (HTTP 422).
EstadoTransições Manuais (PATCH)Pode ser Eliminado?
active suspended não
violated active, suspended não
suspended active sim

Guia Passo a Passo — Cenário Completo

1
Gerir um Alarme Crítico de Serviço
TMF642 POST /alarm  raised → acknowledged → cleared

Cenário: o sistema de monitorização detecta que o serviço vm-alpha-001 está a consumir 97% de CPU. O incidente é registado como alarme, reconhecido pelo operador e resolvido após reinício do processo em causa.

✅
Na interface: use o separador Alarms do módulo Assurance. Os botões Acknowledge e Clear aparecem na linha do alarme conforme o estado actual.
2
Registar uma Medição de Desempenho
TMF628 POST /performanceMeasurement  scheduled → completed

Cenário: registar a latência horária do serviço vm-alpha-001 para análise de tendência. Cada hora, o sistema de monitorização cria uma medição scheduled e, após recolha, preenche o valor e transita para completed.

-- 1. Agendar a medição -- POST /tmf-api/performanceManagement/v4/performanceMeasurement { "name": "Latência horária — vm-alpha 10h00", "service_id": "<uuid-do-serviço>", "metric_name": "latency_ms", "unit_of_measure": "ms", "granularity": "hourly", "scheduled_at": "2026-03-08T10:00:00Z" } -- 2. Após recolha, registar o valor e completar -- PATCH /tmf-api/performanceManagement/v4/performanceMeasurement/<id> { "state": "completed", "metric_value": 12.4, "completed_at": "2026-03-08T10:05:00Z" }
ℹ️
Um valor de metric_value de 12.4 ms está abaixo do threshold de 50 ms do SLO criado no passo anterior — o SLO permaneceria active. Se fosse 78 ms, dispararia uma violação.
3
Criar um SLO e Simular uma Violação
TMF657 POST /serviceLevel  active → violated → active / suspended

Cenário: definir um SLO de latência (<50 ms) para o serviço vm-alpha-001. O sistema de monitorização chama periodicamente check_violations com o valor actual da métrica. Se a latência exceder 55 ms (50 + tolerância 5), o SLO transita automaticamente para violated.

-- 1. Criar o SLO -- POST /tmf-api/serviceLevelManagement/v4/serviceLevel { "name": "Latência SLO < 50ms — vm-alpha", "service_id": "<uuid-do-serviço>", "metric_name": "latency_ms", "threshold_value": 50.0, "direction": "below", "tolerance": 5.0 } → { "id": "uuid", "state": "active" } -- 2. Verificar (latência medida: 78ms — viola o SLO) -- POST /tmf-api/serviceLevelManagement/v4/serviceLevel/check_violations { "service_id": "<uuid>", "metric_name": "latency_ms", "metric_value": 78.0 } → SLO transita para state: "violated" -- 3. Após resolução: repor para active -- PATCH /tmf-api/serviceLevelManagement/v4/serviceLevel/<slo_id> { "state": "active" } -- 4. Suspender durante manutenção -- PATCH /tmf-api/serviceLevelManagement/v4/serviceLevel/<slo_id> { "state": "suspended" }

Resumo — Fluxo Completo de Assurance
Caso de uso típico — monitorização de um serviço vm-alpha em produção
  • 1 Verificar que existe um Service activo no Inventário (TMF638) — obter o service_id
  • 2 Criar um SLO via POST /serviceLevel com a métrica, o threshold e a direcção adequados
  • 3 Periodicamente, criar Medições via POST /performanceMeasurement e preencher o valor com PATCH
  • 4 Ao registar uma medição, chamar POST /serviceLevel/check_violations para actualizar o estado dos SLOs
  • 5 Se um SLO estiver violated, investigar e criar um Alarme via POST /alarm se aplicável
  • 6 Reconhecer o alarme (acknowledged) e resolvê-lo (cleared) após correcção
  • 7 Repor o SLO para active após a situação estar normalizada
  • 8 Eliminar alarmes cleared e medições terminais para manter o dashboard limpo
