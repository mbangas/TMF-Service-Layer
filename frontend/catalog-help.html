❓ Guia de Configuração E2E
Como configurar um serviço do catálogo até à criação de uma ordem
← Voltar ao Catálogo

Sobre este guia

Este guia explica, passo a passo, como configurar um serviço de ponta a ponta (E2E) na TMF Service Layer — desde a criação das especificações no catálogo, passando pelas dependências entre serviços, até ao momento em que é possível lançar uma Ordem de Serviço. O exemplo utilizado é uma VM Completa: um serviço do tipo bundle composto por três serviços elementares.

Cada serviço elementar pode também ser encomendado individualmente, de forma independente.

🗺
Visão Geral — Serviço VM Completa
Arquitetura do bundle e serviços elementares

O serviço VM Completa é um bundle que agrega três serviços elementares. Cada elementar pode ser encomendado de forma autónoma, e o bundle orquestra os três em conjunto.

ℹ️
TMF633 — ServiceSpecification: Cada caixa acima corresponde a uma ServiceSpecification. O bundle está marcado com isBundle: true e referencia os outros três através de serviceSpecRelationship com tipo bundled.
🔀
Dependências de Execução
Ordem de aprovisionamento entre os serviços elementares

Os serviços elementares têm dependências de execução: alguns só podem ser activados depois de outro estar aprovisionado. No caso da VM Completa:

  • A Gestão de Utilizadores e Passwords só pode ser configurada após a VM estar activa no vCenter — não existe SO onde criar os utilizadores antes disso.
  • A Associação de Storage à VM só pode ser executada após a VM estar activa no vCenter — é necessário o identificador da VM para associar o datastore.
  • Não existe dependência directa entre "Gestão de Utilizadores" e "Associação de Storage": podem correr em paralelo após a VM estar pronta.
⚠️
serviceSpecRelationship com tipo "dependency": As dependências são registadas na API TMF633 como relações do tipo dependency na ServiceSpecification dependente (ex: "Gestão de Utilizadores" declara dependência de "VM Elementar"). O motor de orquestração respeita esta ordem ao executar o bundle.

Guia Passo a Passo

1
Criar as Especificações de Serviço Elementares
TMF633 POST /serviceSpecification

Comece por criar individualmente cada um dos três serviços elementares. Cada um é uma ServiceSpecification com isBundle: false. Guarde o id retornado — vai precisar dele nos passos seguintes.

1.1 — VM Elementar (vCenter)

POST /api/v1/serviceSpecification { "name": "VM Elementar vCenter", "description": "Cria e gere o ciclo de vida de uma VM no VMware vCenter", "version": "1.0", "lifecycleStatus": "active", "isBundle": false } → Guardar: spec_vm_id = "uuid-retornado"

1.2 — Gestão de Utilizadores e Passwords

POST /api/v1/serviceSpecification { "name": "Gestão de Utilizadores e Passwords", "description": "Cria e configura utilizadores, passwords e grupos no SO da VM", "version": "1.0", "lifecycleStatus": "active", "isBundle": false } → Guardar: spec_users_id = "uuid-retornado"

1.3 — Associação de Storage à VM

POST /api/v1/serviceSpecification { "name": "Associação de Storage à VM", "description": "Associa e configura volumes de storage (datastore/LUN) à VM", "version": "1.0", "lifecycleStatus": "active", "isBundle": false } → Guardar: spec_storage_id = "uuid-retornado"
✅
Na interface: Catalog → tab "Specifications" → botão "+ New Specification". Preencha o nome, descrição, versão 1.0, status active e deixe "Is Bundle" desmarcado. Repita para cada um dos três serviços.
2
Criar a Especificação Bundle — VM Completa
TMF633 POST /serviceSpecification isBundle: true

Crie agora a especificação bundle. Esta referencia os três elementares através do campo serviceSpecRelationship com tipo bundled.

POST /api/v1/serviceSpecification { "name": "VM Completa", "description": "Bundle: VM vCenter + Gestão de Utilizadores + Associação de Storage", "version": "1.0", "lifecycleStatus": "active", "isBundle": true, "serviceSpecRelationship": [ { "relationshipType": "bundled", "id": "<spec_vm_id>", "name": "VM Elementar vCenter" }, { "relationshipType": "bundled", "id": "<spec_users_id>", "name": "Gestão de Utilizadores e Passwords" }, { "relationshipType": "bundled", "id": "<spec_storage_id>", "name": "Associação de Storage à VM" } ] } → Guardar: spec_bundle_id = "uuid-retornado"
ℹ️
Na interface: marque a caixa "Is Bundle" ao criar a especificação. As relações bundled podem ser adicionadas posteriormente no tab "Spec Dependencies", seleccionando a especificação bundle e adicionando cada elementar com tipo bundled.
3
Definir Características de cada Especificação
TMF633 POST /serviceSpecification/{id}/serviceSpecCharacteristic

As características (ServiceSpecCharacteristic) definem os parâmetros configuráveis de cada serviço — são os campos que o operador preenche ao criar uma ordem.

Características sugeridas — VM Elementar vCenter

Nome Tipo Obrigatório Exemplo
vm_name string sim vm-prod-001
vcpu_count integer sim 4
ram_gb integer sim 8
os_template string (enum) sim ubuntu-22.04 / rhel-9
vcenter_datacenter string sim DC-Lisboa
vcenter_cluster string não Cluster-Prod

Características sugeridas — Gestão de Utilizadores e Passwords

Nome Tipo Obrigatório Exemplo
admin_username string sim svc-admin
password_policy string (enum) sim standard / strict
ldap_integration boolean não true
sudo_groups string não wheel,sysadmin

Características sugeridas — Associação de Storage à VM

Nome Tipo Obrigatório Exemplo
storage_size_gb integer sim 100
storage_type string (enum) sim SSD / HDD / NVMe
datastore_name string sim DS-SSD-Prod
backup_policy string (enum) não daily / weekly / none
✅
Na interface: Catalog → tab "Characteristics", seleccione a especificação no dropdown e clique "+ Add Characteristic" para cada parâmetro acima.
4
Configurar Dependências entre Especificações
TMF633 serviceSpecRelationship type: dependency

Declare que "Gestão de Utilizadores" e "Associação de Storage" dependem de "VM Elementar" estar aprovisionada. Faça um PATCH em cada especificação dependente, adicionando a relação do tipo dependency.

4.1 — Gestão de Utilizadores depende de VM Elementar

PATCH /api/v1/serviceSpecification/<spec_users_id> { "serviceSpecRelationship": [ { "relationshipType": "dependency", "id": "<spec_vm_id>", "name": "VM Elementar vCenter" } ] }

4.2 — Associação de Storage depende de VM Elementar

PATCH /api/v1/serviceSpecification/<spec_storage_id> { "serviceSpecRelationship": [ { "relationshipType": "dependency", "id": "<spec_vm_id>", "name": "VM Elementar vCenter" } ] }
✅
Na interface: Catalog → tab "Spec Dependencies", seleccione "Gestão de Utilizadores e Passwords", clique "+ Add Dependency" e escolha tipo dependency apontando para "VM Elementar vCenter". Repita para "Associação de Storage".
⚠️
Sem dependência circular: Nunca declare que "VM Elementar" depende de si mesma ou crie ciclos. O motor de orquestração detecta ciclos e rejeita a ordem.
5
Criar Catálogo, Categorias e Candidatos
TMF633 /catalog /category /serviceCandidate

Para que uma especificação seja pesquisável e ordenável, precisa de ser publicada num catálogo através de um candidato a serviço (ServiceCandidate). O fluxo é: Catalog → Category → Candidate → Specification.

5.1 — Criar o Catálogo

POST /api/v1/catalog { "name": "Catálogo Infra Privada", "description": "Serviços de infraestrutura virtualizada on-premises", "version": "1.0", "lifecycleStatus": "active" } → Guardar: catalog_id = "uuid-retornado"

5.2 — Criar a Categoria

POST /api/v1/serviceCategory { "name": "Virtualização", "description": "Serviços de computação virtualizada (VMs, contentores)", "version": "1.0", "lifecycleStatus": "active", "isRoot": true } → Guardar: category_id = "uuid-retornado"

5.3 — Criar Candidatos (um por especificação)

Crie um ServiceCandidate para cada especificação que deve aparecer no catálogo. Faça isto para os 4 serviços (3 elementares + bundle).

-- Candidato para o bundle VM Completa -- POST /api/v1/serviceCandidate { "name": "VM Completa", "version": "1.0", "lifecycleStatus": "active", "serviceSpecification": { "id": "<spec_bundle_id>", "name": "VM Completa" }, "category": [{ "id": "<category_id>" }] } -- Repita para cada elementar, substituindo spec_*_id --
✅
Na interface: tab "Catalogs" → create, tab "Categories" → create, tab "Candidates" → create (associe a especificação e a categoria em cada candidato).
6
Criar uma Ordem de Serviço
TMF641 POST /serviceOrder

Com as especificações, características, dependências e candidatos criados, já é possível lançar uma Ordem de Serviço. Pode ordenar o bundle completo (VM Completa) ou qualquer elementar individualmente.

6a — Ordenar o bundle VM Completa (recomendado)

POST /api/v1/serviceOrder { "description": "Provisionar VM Completa para projecto Alpha", "priority": "normal", "orderItem": [ { "id": "1", "action": "add", "service": { "serviceSpecification": { "id": "<spec_bundle_id>" }, "serviceCharacteristic": [ { "name": "vm_name", "value": "vm-alpha-001" }, { "name": "vcpu_count", "value": "4" }, { "name": "ram_gb", "value": "8" }, { "name": "os_template", "value": "ubuntu-22.04"}, { "name": "vcenter_datacenter", "value": "DC-Lisboa" }, { "name": "admin_username", "value": "svc-admin" }, { "name": "password_policy", "value": "strict" }, { "name": "storage_size_gb", "value": "100" }, { "name": "storage_type", "value": "SSD" }, { "name": "datastore_name", "value": "DS-SSD-Prod"} ] } } ] } → O motor de orquestração respeita as dependências: 1º VM Elementar vCenter 2º Gestão de Utilizadores ┐ em paralelo 2º Associação de Storage ┘

6b — Ordenar apenas um elementar (ex: adicionar storage a VM existente)

POST /api/v1/serviceOrder { "description": "Adicionar storage extra à vm-alpha-001", "orderItem": [ { "id": "1", "action": "add", "service": { "serviceSpecification": { "id": "<spec_storage_id>" }, "serviceCharacteristic": [ { "name": "storage_size_gb", "value": "200" }, { "name": "storage_type", "value": "SSD" }, { "name": "datastore_name", "value": "DS-SSD-Prod"} ] } } ] }
ℹ️
Na interface: navegue até Orders → + New Order, seleccione a especificação desejada e preencha as características.
✔
Resumo — Checklist de Configuração E2E
Todos os passos necessários, por ordem
  • 1 Criar especificação VM Elementar vCenter (isBundle: false, status active)
  • 2 Criar especificação Gestão de Utilizadores e Passwords (isBundle: false)
  • 3 Criar especificação Associação de Storage à VM (isBundle: false)
  • 4 Criar especificação VM Completa (isBundle: true) com relações bundled para os 3 elementares
  • 5 Adicionar características a cada uma das 4 especificações (tab "Characteristics")
  • 6 Declarar dependência de "Gestão de Utilizadores" → "VM Elementar" (tipo dependency)
  • 7 Declarar dependência de "Associação de Storage" → "VM Elementar" (tipo dependency)
  • 8 Criar Catálogo "Infra Privada"
  • 9 Criar Categoria "Virtualização" (root) e associar ao catálogo
  • 10 Criar Candidatos para cada uma das 4 especificações, apontando para a categoria
  • 11 Navegar a Orders → + New Order e seleccionar "VM Completa" ou qualquer elementar
