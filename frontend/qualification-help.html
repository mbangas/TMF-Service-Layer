Como criar e gerir pedidos de qualificação — TMF645 Service Qualification Management
← Voltar à Qualificação

Sobre este guia

Este guia explica como usar o módulo de Qualificação de Serviços da TMF Service Layer (TMF645 — Service Qualification Management). A qualificação é uma verificação de viabilidade pré-venda que determina se um serviço pode ser entregue a um cliente antes de criar uma ordem formal.

Um pedido de qualificação (ServiceQualification) contém um ou mais itens de qualificação (ServiceQualificationItem), cada um referenciando uma especificação de serviço do catálogo e devolvendo um resultado: approved, rejected ou unableToProvide.

Após a qualificação ser accepted, o resultado serve de fundamento para criar uma Ordem de Serviço (TMF641). Consulte o Guia de Ordens para o passo seguinte.

⚠
Pré-requisito — Catálogo Configurado (recomendado)
Os itens de qualificação podem referenciar especificações de serviço do catálogo (TMF633)

Cada ServiceQualificationItem aceita um service_spec_id (UUID de uma ServiceSpecification do catálogo). Este campo é opcional: pode criar itens sem referenciar o catálogo, mas a referência enriquece o resultado e permite ligar a qualificação ao processo de encomenda subsequente.

ℹ️
Ainda não tem o catálogo configurado? Use a secção Gerar Dados de Exemplo no final desta página para criar automaticamente uma qualificação de teste. O catálogo não é obrigatório — pode omitir o service_spec_id.
🗺
Estrutura de uma ServiceQualification
Campos, itens de qualificação e ciclo de vida

Uma ServiceQualification agrupa um ou mais itens (ServiceQualificationItem). Cada item representa a verificação de viabilidade de um serviço específico e tem o seu próprio resultado. O estado do pedido evolui de forma independente dos resultados dos itens.

Campos de cada ServiceQualificationItem

Resultados possíveis de um item

Estado do Item Significado Acção Típica

rejected Serviço não pode ser entregue nas condições pedidas Verificar termination_error e ajustar requisitos
unableToProvide Impossível determinar a viabilidade (dados insuficientes, timeout, etc.) Reiterar o pedido com mais informação ou escalar

Ciclo de Vida da Qualificação
Transições de estado e regras de negócio

Uma qualificação começa sempre no estado acknowledged (criada pelo sistema). O processamento avança por transições explícitas via PATCH. Os estados terminais (accepted, rejected, cancelled) não permitem mais transições.

Estado Actual Transições Permitidas Pode ser Eliminada?
acknowledged inProgress, cancelled não
inProgress accepted, rejected, cancelled não
accepted — (terminal) sim
rejected — (terminal) sim
cancelled — (terminal) sim

Guia Passo a Passo

1
Criar um Pedido de Qualificação
TMF645 POST /checkServiceQualification

O pedido é criado com o estado acknowledged automaticamente. O exemplo abaixo qualifica o serviço VM Completa para um cliente do Projecto Alpha, verificando disponibilidade no DC Lisboa.

1a — Qualificação com referência ao catálogo (recomendado)

Inclua o service_spec_id para ligar o pedido à especificação do catálogo. O qualifier_message documenta o resultado para o cliente.

POST /tmf-api/serviceQualificationManagement/v4/checkServiceQualification { "name": "Qualificação VM Completa — Cliente Projecto Alpha", "description": "Verificação de viabilidade pré-venda para DC Lisboa", "expected_qualification_date": "2026-04-01T12:00:00Z", "expiration_date": "2026-06-30T23:59:59Z", "items": [ { "service_spec_id": "<uuid-da-spec-vm-completa>", "state": "approved", "qualifier_message": "Capacidade disponível em DC Lisboa — entrega em 5 dias úteis" } ] } → Resposta: { "id": "uuid", "state": "acknowledged", "items": [ … ] }

1b — Qualificação sem catálogo (mínimo)

Pode criar um pedido sem especificações de catálogo — útil para verificações genéricas ou quando o catálogo ainda não está configurado.

POST /tmf-api/serviceQualificationManagement/v4/checkServiceQualification { "name": "Verificação genérica — Cliente Beta", "description": "Qualificação exploratória sem especificação definida", "items": [ { "state": "approved", "qualifier_message": "Serviço aprovado para entrega" } ] }

1c — Multiplos serviços na mesma qualificação

Pode incluir vários itens numa única qualificação — um por cada serviço a verificar.

POST /tmf-api/serviceQualificationManagement/v4/checkServiceQualification { "name": "Qualificação Bundle VM — DC Lisboa e DC Porto", "items": [ { "service_spec_id": "<uuid-vm-completa>", "state": "approved", "qualifier_message": "Disponível em DC Lisboa" }, { "service_spec_id": "<uuid-storage>", "state": "rejected", "qualifier_message": "Storage SSD indisponível em DC Porto", "termination_error": "Capacidade de storage esgotada neste DC" } ] }
2
Avançar o Estado do Pedido
TMF645 PATCH /checkServiceQualification/{id}

Use PATCH para fazer transitar o estado do pedido. Apenas o campo state (e os campos de metadados) pode ser alterado após a criação.

2a — Iniciar análise (acknowledged → inProgress)

PATCH /tmf-api/serviceQualificationManagement/v4/checkServiceQualification/{id} { "state": "inProgress" } → Resposta: { "state": "inProgress", … }

2b — Aceitar a qualificação (inProgress → accepted)

Um pedido accepted confirma que todos os serviços avaliados podem ser entregues. Serve de aprovação formal para emitir a Ordem de Serviço.

PATCH /tmf-api/serviceQualificationManagement/v4/checkServiceQualification/{id} { "state": "accepted" } → Estado terminal. Qualificação pode agora ser eliminada.

2c — Rejeitar (inProgress → rejected)

PATCH /tmf-api/serviceQualificationManagement/v4/checkServiceQualification/{id} { "state": "rejected" } → Estado terminal. Ordem de serviço não deve ser criada.

2d — Cancelar (acknowledged ou inProgress → cancelled)

PATCH /tmf-api/serviceQualificationManagement/v4/checkServiceQualification/{id} { "state": "cancelled" }
⚠️
Estados terminais: após accepted, rejected ou cancelled, o estado não pode mais ser alterado. Crie um novo pedido de qualificação se necessário.
3
Consultar e Filtrar Qualificações
TMF645 GET /checkServiceQualification

Listar todas as qualificações (paginado)

GET /tmf-api/serviceQualificationManagement/v4/checkServiceQualification?limit=20&offset=0 → Headers: X-Total-Count, X-Result-Count

Filtrar por estado

GET /tmf-api/serviceQualificationManagement/v4/checkServiceQualification?state=inProgress → Apenas qualificações em análise

Obter uma qualificação específica

GET /tmf-api/serviceQualificationManagement/v4/checkServiceQualification/{id} → Inclui os itens de qualificação (items[]) na resposta
ℹ️
Os cabeçalhos X-Total-Count e X-Result-Count devolvidos na listagem seguem o padrão TMF de paginação. Use offset + limit para navegar entre páginas.
4
Eliminar uma Qualificação
TMF645 DELETE /checkServiceQualification/{id}

Só é permitido eliminar qualificações em estados terminais: accepted, rejected ou cancelled. Tentativas de eliminar qualificações em acknowledged ou inProgress devolvem HTTP 409 Conflict.

DELETE /tmf-api/serviceQualificationManagement/v4/checkServiceQualification/{id} → 204 No Content (sucesso) → 409 Conflict se estado não for terminal
✔
Resumo do Fluxo de Qualificação
Caso de uso típico — Qualificação da VM Completa para cliente pré-venda
  • 1 (Opcional) Confirmar que a ServiceSpecification "VM Completa" existe no catálogo
  • 2 Criar a qualificação via POST /checkServiceQualification com os itens desejados
  • 3 Pedido criado automaticamente no estado acknowledged
  • 4 Avançar para inProgress via PATCH enquanto se verifica a viabilidade
  • 5 Após análise, transitar para accepted ou rejected
  • 6 Se accepted → criar Ordem de Serviço (TMF641) com base no resultado
  • 7 Após a ordem estar criada, eliminar a qualificação (estado terminal)
