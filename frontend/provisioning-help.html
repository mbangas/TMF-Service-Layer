Como criar e gerir jobs de activação — TMF640 Service Activation & Configuration
← Voltar ao Aprovisionamento

Sobre este guia

Este guia explica como usar o módulo de Aprovisionamento da TMF Service Layer (TMF640 — Service Activation & Configuration). O módulo gere o ciclo de vida activo dos serviços através de jobs de activação (ServiceActivationJob), que operam sobre instâncias de serviço existentes no inventário (TMF638).

Existem cinco tipos de job: provision, activate, modify, deactivate e terminate. Cada tipo aplica uma acção diferente sobre o serviço alvo e modifica o seu estado no inventário.

O inventário deve ter uma instância de serviço antes de se poder criar um job. Consulte o Guia de Ordens para perceber como as ordens criam entradas no inventário, ou crie um serviço directamente via Inventário.

⚠
Pré-requisito — Serviço no Inventário
O serviço alvo deve existir no inventário (TMF638) antes de criar um job

Para criar um ServiceActivationJob é necessário que exista uma instância de Service no inventário com o estado adequado ao tipo de job que se pretende executar:

Tipo de Job Estado do Serviço Exigido Estado após Sucesso Descrição
provision inactive active Aprovisionamento inicial do serviço
activate inactive active Re-activação de serviço previamente desactivado
modify active active Modificação de parâmetros de configuração
deactivate active inactive Suspensão temporária do serviço
terminate active ou inactive terminated Encerramento definitivo — irreversível
ℹ️
Ainda não tem serviços no inventário? Use a secção Gerar Dados de Exemplo no final desta página para criar automaticamente um serviço de teste e os respectivos jobs.
🗺
Estrutura de um ServiceActivationJob
Campos, parâmetros de configuração e relacionamento com o inventário

Um ServiceActivationJob encapsula uma operação de activação sobre um serviço. Além dos campos de controlo, pode conter uma lista de parâmetros de configuração (ServiceConfigurationParam) que são aplicados ao serviço quando o job termina com sucesso.

Campos do ServiceActivationJob

Campo Obrigatório Tipo Descrição
namesimstringNome descritivo do job (ex: "Provision VM Alpha")
job_typesimenumprovision | activate | modify | deactivate | terminate
service_idsimuuidID do serviço alvo no inventário (TMF638)
descriptionnãostringDescrição livre do job
modenãostringModo de execução: immediate ou deferred
start_modenãostringInício automático ou manual: automatic | manual
scheduled_start_datenãodatetimeData/hora de início programado (modo deferred)
scheduled_completion_datenãodatetimeData/hora limite de conclusão pretendida
paramsnãoarrayLista de parâmetros de configuração (ver abaixo)

Campos de cada ServiceConfigurationParam

Campo Obrigatório Tipo Descrição
namesimstringNome do parâmetro (ex: "speed_mbps", "vcpu_count")
valuenãostringValor do parâmetro (sempre serializado como string)
value_typenãostringTipo de dados: "string", "integer", "boolean", etc.
🔄
Ciclo de Vida do Job — Transições de Estado
accepted → running → succeeded / failed / cancelled

Após a criação, o job encontra-se no estado accepted. As transições de estado são efectuadas via PATCH na API ou pelos botões de acção na interface.

Transições via API

-- Iniciar execução do job -- PATCH /tmf-api/serviceActivationConfiguration/v4/serviceActivationJob/<job_id> { "state": "running" } -- Marcar como sucesso (actualiza estado do serviço no inventário) -- PATCH /tmf-api/serviceActivationConfiguration/v4/serviceActivationJob/<job_id> { "state": "succeeded" } -- Marcar como falha (com mensagem de erro) -- PATCH /tmf-api/serviceActivationConfiguration/v4/serviceActivationJob/<job_id> { "state": "failed", "error_message": "Timeout ao contactar vCenter" } -- Cancelar (de accepted ou running) -- PATCH /tmf-api/serviceActivationConfiguration/v4/serviceActivationJob/<job_id> { "state": "cancelled" }
✅
Na interface: use os botões Run, Succeed, Fail e Cancel na linha de cada job. Os botões disponíveis mudam automaticamente conforme o estado actual.
⚠️
Apenas jobs nos estados failed ou cancelled podem ser eliminados. Para apagar um job, coloque-o num estado terminal primeiro.

Guia por Casos de Uso

1
Caso 1 — Provision: Aprovisionamento Inicial provision
TMF640 POST /serviceActivationJob  Serviço: inactive → active

O job do tipo provision é usado para o aprovisionamento inicial de um serviço: configura e activa um serviço que ainda não foi aprovisionado. O serviço alvo deve estar no estado inactive. Ao transitar para succeeded, o estado do serviço passa automaticamente a active.

Criar um job de provision

-- 1. Verificar que o serviço existe e está inactive -- GET /tmf-api/serviceInventory/v4/service/<service_id> → { "id": "uuid", "state": "inactive", "name": "vm-alpha-001" } -- 2. Criar o job de provision -- POST /tmf-api/serviceActivationConfiguration/v4/serviceActivationJob { "name": "Provision VM Alpha", "description": "Aprovisionamento inicial da VM vm-alpha-001 no vCenter DC-Lisboa", "job_type": "provision", "service_id": "<uuid-do-serviço>", "params": [ { "name": "vcpu_count", "value": "4", "value_type": "integer" }, { "name": "ram_gb", "value": "8", "value_type": "integer" }, { "name": "os_template", "value": "ubuntu-22.04", "value_type": "string" }, { "name": "vcenter_datacenter","value": "DC-Lisboa", "value_type": "string" } ] } → Resposta: { "id": "uuid-job", "state": "accepted", "job_type": "provision", … } -- 3. Iniciar a execução -- PATCH /tmf-api/serviceActivationConfiguration/v4/serviceActivationJob/<job_id> { "state": "running" } -- 4. Marcar como sucesso (serviço passa a active) -- PATCH /tmf-api/serviceActivationConfiguration/v4/serviceActivationJob/<job_id> { "state": "succeeded" } → Serviço no inventário actualizado: state: "active"
✅
Na interface: Provisioning → botão "+ New Job". Seleccione o serviço alvo, escolha tipo provision, e adicione os parâmetros de configuração. O job é criado no estado accepted. Use os botões Run e Succeed para avançar no ciclo de vida.
2
Caso 2 — Activate: Re-activação de Serviço activate
TMF640 POST /serviceActivationJob  Serviço: inactive → active

O job do tipo activate é usado para re-activar um serviço que foi previamente desactivado (sem ser terminado). É semelhante ao provision no que respeita à pré-condição e ao resultado, mas o significado operacional é diferente: o serviço já existe e foi apenas suspenso.

Pré-condição: serviço no estado inactive (tipicamente após um deactivate bem sucedido).

POST /tmf-api/serviceActivationConfiguration/v4/serviceActivationJob { "name": "Activar VM Alpha — Retorno ao Serviço", "description": "Reactivação após janela de manutenção", "job_type": "activate", "service_id": "<uuid-do-serviço>" } → Resposta: { "state": "accepted", "job_type": "activate" } -- Progressão: accepted → running → succeeded (serviço volta a active) --
ℹ️
Diferença entre activate e provision: O provision configura o serviço de raiz (instala, cria recursos). O activate apenas retoma um serviço já configurado que estava suspenso (ex: ligar novamente uma VM que foi desligada).
3
Caso 3 — Modify: Modificação de Configuração modify
TMF640 POST /serviceActivationJob  Serviço: active → active (com nova configuração)

O job do tipo modify permite alterar parâmetros de configuração de um serviço que já está activo — sem o desactivar. Típico quando se precisa de aumentar recursos de uma VM (ex: mais vCPUs ou RAM) em hot-add, ou alterar uma política de backup.

Pré-condição: serviço no estado active. O estado permanece active após o sucesso.

POST /tmf-api/serviceActivationConfiguration/v4/serviceActivationJob { "name": "Upgrade VM Alpha — 4 vCPU para 8 vCPU", "description": "Expansão de recursos para pico de carga Q1", "job_type": "modify", "service_id": "<uuid-do-serviço>", "params": [ { "name": "vcpu_count", "value": "8", "value_type": "integer" }, { "name": "ram_gb", "value": "16", "value_type": "integer" } ] }
⚠️
O job modify requer que o serviço esteja active. Se tentar criar um job modify sobre um serviço inactive, a API devolve um erro 422.
4
Caso 4 — Deactivate: Suspensão Temporária deactivate
TMF640 POST /serviceActivationJob  Serviço: active → inactive

O job do tipo deactivate suspende temporariamente um serviço activo, colocando-o no estado inactive. É uma operação reversível: o serviço pode ser re-activado posteriormente com um job activate.

Casos de uso típicos: janelas de manutenção, suspensão de serviços de clientes em incumprimento, poupança de recursos em períodos de baixo tráfego.

POST /tmf-api/serviceActivationConfiguration/v4/serviceActivationJob { "name": "Desactivar VM Alpha — Janela de Manutenção", "description": "Suspensão durante janela de manutenção 03:00-05:00", "job_type": "deactivate", "service_id": "<uuid-do-serviço>", "scheduled_start_date": "2026-03-15T03:00:00Z", "scheduled_completion_date": "2026-03-15T05:00:00Z" } → Após succeeded: serviço no inventário com state: "inactive"
ℹ️
Após o deactivate bem sucedido, o serviço fica inactive no inventário. Para o voltar a activar, crie um job activate (não é necessário um novo provision).
5
Caso 5 — Terminate: Encerramento Definitivo terminate
TMF640 POST /serviceActivationJob  Serviço: active ou inactive → terminated

O job do tipo terminate procede ao encerramento definitivo e irreversível do serviço — destrói os recursos associados (apaga a VM, os volumes, as configurações de utilizadores). O serviço passa para o estado terminated no inventário.

Ao contrário do deactivate, o terminate não pode ser desfeito. É aceite tanto a partir de um serviço active como inactive.

POST /tmf-api/serviceActivationConfiguration/v4/serviceActivationJob { "name": "Terminar VM Alpha — Projecto Encerrado", "description": "Encerramento definitivo e remoção de todos os recursos do projecto Alpha", "job_type": "terminate", "service_id": "<uuid-do-serviço>" } → Após succeeded: serviço no inventário com state: "terminated"
⚠️
Atenção: operação irreversível. Um serviço no estado terminated não pode ser re-activado. Para voltar a ter o serviço disponível, seria necessário criar um novo serviço no inventário e executar um novo provision.
6
Listar e Filtrar Jobs
TMF640 GET /serviceActivationJob

A API suporta filtragem e paginação na listagem de jobs. Use os parâmetros de query para encontrar jobs por estado, tipo ou serviço alvo.

-- Listar todos os jobs (paginação por omissão: offset=0, limit=20) -- GET /tmf-api/serviceActivationConfiguration/v4/serviceActivationJob -- Filtrar por estado -- GET /tmf-api/serviceActivationConfiguration/v4/serviceActivationJob?state=accepted GET /tmf-api/serviceActivationConfiguration/v4/serviceActivationJob?state=running GET /tmf-api/serviceActivationConfiguration/v4/serviceActivationJob?state=succeeded -- Filtrar por tipo de job -- GET /tmf-api/serviceActivationConfiguration/v4/serviceActivationJob?job_type=provision -- Filtrar por serviço alvo -- GET /tmf-api/serviceActivationConfiguration/v4/serviceActivationJob?service_id=<uuid> -- Paginação -- GET /tmf-api/serviceActivationConfiguration/v4/serviceActivationJob?offset=20&limit=10 → Headers de resposta: X-Total-Count (total), X-Result-Count (nesta página) -- Obter um job específico -- GET /tmf-api/serviceActivationConfiguration/v4/serviceActivationJob/<job_id> -- Eliminar job (só failed ou cancelled) -- DELETE /tmf-api/serviceActivationConfiguration/v4/serviceActivationJob/<job_id>
✅
Na interface: use os dropdowns de filtro no topo da tabela de jobs para filtrar por estado (All states, accepted, running, etc.) e por tipo (job type). A paginação é gerida automaticamente pela interface.
✔
Resumo — Fluxo Completo de Aprovisionamento
Todos os tipos de job e o seu impacto no estado do serviço
  • 1 Verificar que existe um Service no Inventário (TMF638) com o estado adequado ao job
  • 2 Navegar a Provisioning → + New Job
  • 3 Seleccionar o service_id do serviço alvo
  • 4 Escolher o job_type: provision / activate / modify / deactivate / terminate
  • 5 Adicionar os parâmetros de configuração relevantes (params)
  • 6 Clicar "Create Job" — job criado no estado accepted
  • 7 Clicar Run para transitar para running
  • 8 Após execução bem sucedida, clicar Succeed → succeeded
  • 9 O estado do serviço no Inventário é actualizado automaticamente

Gerar Dados de Teste
Recria automaticamente o cenário de aprovisionamento da VM Alpha — mesmo caso de teste utilizado nos testes de integração

Clique no botão abaixo para gerar automaticamente os dados de teste de aprovisionamento usados nos testes de integração TMF640. O script executa os seguintes passos:

  1. Procura um serviço vm-alpha-001 no inventário; se não existir, cria-o no estado inactive
  2. Cria um job provision — "Provision VM Alpha" — com parâmetros de configuração (vcpu, ram, os_template, datacenter)
  3. Avança o job para running
  4. Marca o job como succeeded → o serviço passa a active
  5. Cria um job modify — "Upgrade VM Alpha" — para aumentar vCPUs de 4 para 8
  6. Cria um job deactivate — em estado accepted (para demonstração do ciclo)
⚠️
Execuções repetidas: Se clicar mais do que uma vez, os serviços são criados com sufixo numérico (ex: "vm-alpha-001-2"). Cada execução cria um conjunto de dados independente.
